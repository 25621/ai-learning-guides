# 探索 PettingZoo 環境 🦓

## 什麼是 PettingZoo？

如果你曾經做過單代理人強化學習，你可能用過 **Gymnasium**（OpenAI Gym 的繼承者）。每個環境看起來都一樣：`env.reset()`，`env.step(action) → obs, reward, done, info` —— 一個對世界的新*觀察*，一個純量*獎勵*信號，一個表示「遊戲結束」的*完成*標誌，以及一個用於調試額外內容的*資訊*字典。這種統一性正是強化學習庫能夠發揮作用的原因。

**PettingZoo** 的理念完全相同，但它是針對*多代理人*的。它是一個多代理人環境的動物園 —— 全部遵循一個定義良好的 API：
- **經典玩具問題**：像「石頭剪刀布」這樣的簡單環境，用於測試基礎演算法。
- **協同網格世界**：代理人在網格中導航以實現共同目標。
- **雅達利多人遊戲**：經典的競技遊戲，如 Pong。
- **MPE (多粒子環境)**：用於複雜協調和競爭的連續空間物理環境。

如果你能編寫一段在某個 PettingZoo 環境下運行的代碼，你幾乎不需要任何修改就可以將其插入到任何其他環境中。

## 兩種 API 風格

多代理人設置比單代理人設置更複雜，因為兩個代理人可以同時採取行動，或者輪流採取行動，甚至按任意順序採取行動。PettingZoo 透過兩種並行的 API 解決了這個問題：

### 1) AEC (代理人-環境循環 Agent-Environment-Cycle)

一次只有一個代理人採取行動。環境按某種順序循環遍歷代理人，每個代理人都會得到：
- **觀察 (observation)** —— 他們*現在*看到了什麼，
- **獎勵 (reward)** —— 在上一輪完整環節中透過*聯合*動作賺取的收益（也就是說，這是*所有*代理人行動的結果，而不僅僅是你；例如在國際象棋比賽中，你的獎勵反映的是你對手最後一步棋之後的棋盤狀態，而不僅僅是你的），
- **終止標誌 (termination flag)** —— 當回合*自然*結束（如：將軍、有人獲勝）時為 `True`，
- **截斷標誌 (truncation flag)** —— 當回合因時間限制而*提前*結束時為 `True`。

這對於像國際象棋、圍棋、撲克這樣的**回合制遊戲**來說非常自然。

```python
env.reset()
for agent in env.agent_iter():
    obs, reward, term, trunc, info = env.last()
    if term or trunc:
        env.step(None)
        continue
    action = my_policy(obs, agent)
    env.step(action)
```

### 2) Parallel (並行)

所有代理人在每一步都同時觀察並採取行動。`step()` 接收一個動作*字典*並返回觀察字典和獎勵字典。

這對於像 MPE（多粒子環境，所有點狀代理人同時移動）或多代理人網格世界這樣的**實時遊戲**來說非常自然。

```python
obs, info = env.reset()
while env.agents:
    actions = {a: my_policy(obs[a]) for a in env.agents}
    obs, rewards, terms, truncs, info = env.step(actions)
```

這兩種風格是**同構的 (isomorphic)** —— 結構上等價且可以相互轉換：任何 AEC 環境都可以自動包裝成並行環境，反之亦然。PettingZoo 提供了轉換包裝器，因此你只需要針對一種風格編寫代碼。

## 現實生活中的類比

- **AEC = 棋盤遊戲之夜**。「Alice 行動。現在是 Bob。現在是 Carol。回到 Alice。」下一個行動的人會看到最新的棋盤狀態。
- **Parallel = 多人視頻遊戲**。所有四名玩家同時按下按鈕；遊戲每秒更新世界 60 次。
- **為什麼統一的 API 很重要**。想像一下，如果每款多人視頻遊戲都需要自己的搖桿。PettingZoo 就是多代理人強化學習 (MARL) 的「通用搖桿」。

## 我們的代碼做了什麼

我們從頭開始構建了一個 **PettingZoo 風格**的環境：**迭代協同博弈 (Iterated Coordination Game)**。兩個代理人反覆選擇通道 `0` 或 `1`：

- 選擇相同 → 兩者都獲得 +1
- 選擇不同 → 兩者都獲得 -1

每個代理人收到的**觀察**是之前的*聯合動作* —— 兩個代理人上一輪選擇的內容，並打包成一個整數。具體來說：每個代理人最後的動作是 `{start, 0, 1}`（3 種狀態）之一，因此這對動作編碼為 `3 × agent_1_state + agent_2_state`，產生 9 個可能的整數（0 – 8）。整數 0 是「開始」狀態 —— 它表示尚未採取任何動作（一個回合的最開始）。
一個回合持續 25 步，因此每個代理人的最大總回報為 +25，最小總回報為 -25。**隨機遊戲的得分 ≈ 0**，因為在每一步，兩個獨立的隨機代理人以相同的概率各選擇 0 或 1：它們有 50% 的時間匹配 (+1)，50% 的時間不匹配 (-1)，預期的單步獎勵為 0.5 × (+1) + 0.5 × (-1) = **0**。25 步累加後，預期的回合回報也是 0。

然後我們：

1. **透過隨機展開演示 AEC 介面** —— 這確認了基本的 AEC 循環：`agent_iter()` 產生當前輪到的代理人，`last()` 讀取該代理人當前的觀察和累積獎勵，`step()` 將他們選擇的動作交還給環境。
2. **透過並行介面訓練兩個獨立的 Q 學習器**。每個代理人都維護自己的 Q 表，以**聯合動作觀察**（編碼了*兩個*代理人上一輪行為的單個整數）作為鍵，因此它可以學習「當上次我們都選 0 時，我這次也應該選 0」。
3. **嘗試導入真實的 `pettingzoo` 庫**並使用隨機策略展開其內置環境之一（石頭剪刀布）。如果未安裝 PettingZoo，我們將跳過此步驟並顯示一條友好的消息。

### 你應該看到的結果

| 階段 | 預期結果 |
|-------|----------|
| 隨機展開 (AEC)            | 平均回合回報接近 **0** —— 隨機代理人獨立選擇通道，匹配和不匹配的情況大致相等。 |
| 獨立 Q 學習器 (並行) —— 前 100 個回合 | 約 **0** —— 代理人在探索時仍然表現出很大程度的隨機性。 |
| 獨立 Q 學習器 —— 最後 100 個回合             | 強正值，**+20 到 +25** —— **協同已經出現**：兩個代理人都學會了在每一輪可靠地選擇相同的通道。 |

## 為什麼要先構建自定義環境？

因為 **API 本身就是課程。**（理解如何構建多個代理人與環境之間的交互結構比具體的遊戲規則更重要。）多代理人強化學習有許多形式（回合制、實時、協同、競技、混合），它們都符合 AEC / 並行模式。一旦你實現了這兩個循環，每個 PettingZoo 環境都只是插入不同的環境構造函數的問題 —— 訓練代碼保持不變。

這正是 Gymnasium 改變單代理人強化學習的方式：透過將環境變成統一介面背後的黑盒。

## 獨立 Q 學習在何處有利有弊

協同博弈是*寬容的* —— 代理人共享獎勵符號，因此他們的利益是一致的。獨立學習器可以愉快地解決這個問題，因為一個代理人的任何改進都會幫助另一個代理人。

在**對抗性**遊戲 (RPS) 中，獨立 Q 學習會永遠振盪（當一個代理人適應時，另一個代理人會改變策略進行反擊，導致無休止的追逐）。
在**部分可觀測 (partially-observable)** 的遊戲中，它根本無法學習，因為「觀察」只是狀態的一部分（一個代理人可能會因為一個好的動作而受到懲罰，僅僅是因為它看不到另一個代理人在做什麼）。PettingZoo 包含這兩種環境，因此你可以親眼看到這些失敗模式。

## 需要記住的關鍵術語

| 單詞 | 含義 |
|------|---------|
| **PettingZoo**     | 多代理人強化學習的 Gymnasium —— 一個標準化多代理人強化學習環境庫 |
| **AEC**            | 代理人-環境循環：每步一個代理人行動（回合制） |
| **Parallel API**   | 所有代理人在每步同時行動 |
| **MPE**            | 多粒子環境，PettingZoo 附帶的一個流行的協同/競爭測試平台（通常涉及移動的點執行基於物理的任務）。 |
| **CTDE**           | 集中式訓練，分佈式執行 (Centralised Training, Decentralised Execution) —— 以全局視野（訪問所有狀態）進行訓練，以局部觀察（每個代理人根據其有限的視野行動）進行部署。 |
| **獨立 Q 學習 (Independent Q-learning)** | 每個代理人運行普通的 Q 學習（標準、未經修改的 Q 學習演算法），忽略其他學習者的存在。 |

---

## 一句話總結

> **PettingZoo 為每個多代理人環境提供了相同的形狀 —— 因此你今天編寫的代碼在明天完全不同的遊戲中仍然適用。**

一旦這兩種 API 風格變成你的第二本能，你就可以進階到 MADDPG（連續控制代理人的集中式評論家）、QMIX（協同團隊的值混合）、MAPPO（多代理人 PPO）或任何其他現代多代理人強化學習演算法 —— 你代碼中的環境部分永遠不需要改變。
