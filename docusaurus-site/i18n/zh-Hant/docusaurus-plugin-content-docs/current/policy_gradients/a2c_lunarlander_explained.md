# A2C：演員與評論家的協同工作

## 核心思想

REINFORCE 演算法必須等到遊戲完全結束後才進行更新。這就像是一個教練在整場足球比賽中保持沉默，直到比賽結束才給出所有回饋。

**A2C (Advantage Actor-Critic)** 在遊戲進行過程中給出回饋——每隔幾步，教練就會暫停並說：「那次傳球很棒！那次鏟球很糟糕！」

這使得學習速度更快且更有效率。

---

## 認識兩個角色

> **什麼是 LunarLander？** 在本文檔中，我們使用 **LunarLander** 環境——這是一個物理模擬環境，你需要控制一艘小型航天器，使用三個引擎（左側、主引擎和右側）使其平穩降落在月球的目標著陸點上。它是強化學習中的一個標準基準，可在 Gymnasium 庫中找到。

### 演員 (Actor) 🎭
**演員**就是策略（Policy）——它決定採取哪種行動。

> 「我處於這種狀態。我應該啟動左引擎還是右引擎？」

**現實生活中的例子：** 汽車的 *駕駛員*，負責轉動方向盤和踩踏板。

### 評論家 (Critic) 🎬
**評論家**負責估算目前情況的好壞——即價值 V(s)。

> 「處於這種狀態大約值 +150 分的未來總獎勵。」

**現實生活中的例子：** 坐在駕駛員旁邊的 *導航員*，會說：「我們走在一條好路上——預計 30 分鐘後到達。」 或者 「我們要進入擁堵路段了——這會很慢。」

### 他們共享一個大腦
在我們的實現中，兩者使用 **相同的神經網路主幹**：

```
          狀態 (LunarLander 為 8 個數字)
                       ↓
          ┌─────────────────────────┐
          │  共享層 (Shared Layers)  │
          │  [256 個神經元] → ReLU   │
          │  [256 個神經元] → ReLU   │
          └────────┬────────┬───────┘
                   ↓        ↓
          演員頭 (Actor Head)  評論家頭 (Critic Head)
          [4 個輸出]            [1 個輸出]
          (動作機率)            (V(s))
```

- **ReLU** (修正線性單元)：每層之後應用的啟動函數——它輸出 `max(0, x)`，保留正值並將負值置為零。這使得網路能夠學習非線性模式。
- **動作機率 (action probs)**：採取 4 種動作中每一種的機率。演員每一步都會從這個分佈中採樣來選擇一個動作。

**現實生活中的例子：** 一個大腦，兩份工作——就像一個既能開車（演員）又知道路線好壞（評論家）的出租車司機。共享大腦意味著學習速度更快！

## 優勢 (Advantage)：這比預期的好嗎？

就像帶有基準的 REINFORCE 一樣，A2C 計算 **優勢 (Advantage)**：

> A(s, a) = 「實際結果」 − 「我們的預期」

但在 A2C 中，「實際結果」來自評論家的 **n 步自舉 (n-step bootstrap)**（**自舉 (bootstrapping)** = 使用評論家自己的預測 V(s) 來近似未來步數的價值，而不是等待實際回合結束——這就像在學期中利用目前的成績來預估期末考試分數）：

```
實際 TD 回報: r_t + γ · r_{t+1} + γ² · r_{t+2} + ... + γⁿ · V(s_{t+n})
優勢 A_t = TD 回報 - V(s_t)
```

**現實生活中的例子：** 你預期在這場比賽中進 3 個球 (V(s))。如果你進了 5 個球，你的優勢就是 +2。如果你只進了 1 個球，你的優勢就是 -2。

正優勢 → 「那個動作比預期的更有幫助 → 多做這個動作！」
負優勢 → 「那個動作的幫助比預期的少 → 少做這個動作！」

## 為什麼使用多個平行環境？

我們的 A2C 使用 **8 個副本** 的 LunarLander 同時運行！

**為什麼？** 因為來自單個環境的經歷是 **相關的 (correlated)**——每一步都緊隨前一步。這種相關性會誤導神經網路，使其認為某些模式比實際更可靠。

透過 8 個環境，每一步都能提供來自完全不同情況的 8 個獨立經歷。這打破了相關性，並顯著穩定了訓練過程。

**現實生活中的例子：** 為了了解天氣，哪種方式更好：
- 連續 8 小時觀察一個城市（相關的——如果下午 2 點是晴天，下午 3 點可能也是晴天）
- 同時觀察 8 個城市（去相關的——不同的天氣模式，更多的資訊！）

```
環境 1: [降落在月球上, 啟動左引擎, 墜毀, 重置...]
環境 2: [下降太快, 同時啟動兩個引擎, 懸停, 著陸...]
環境 3: [向右傾斜, 啟動右引擎, 穩定, 著陸...]
...
環境 8: [向左漂移, 啟動左引擎, 平穩, ...]
```

所有 8 個環境同時更新網路——每次更新可獲得 8 倍多樣化的經歷！

## N 步更新：不要等到遊戲結束

REINFORCE 等待完整的一個回合（可能是 1000 步！）。

A2C 每隔 **n_steps = 128 步** 進行一次更新：

```
在 8 個環境中運行 128 步
    → 獲得 128 × 8 = 1024 個經歷元組
    → 計算優勢和回報
    → 更新演員和評論家
    → 再運行 128 步...
```

**現實生活中的例子：** 一個學生在為考試複習。
- REINFORCE 風格：讀完整個教科書，然後做模擬測試。
- A2C 風格：讀 10 頁，做個小測驗，再讀 10 頁，再做個測驗...

更頻繁的回饋 = 更快的學習！

## 三種損失結合

A2C 同時使用三個損失項進行訓練：

**損失 (Loss)** 是優化器試圖最小化的數字。損失越小意味著網路目前的行為越接近訓練目標。

### 1. 演員損失 (策略梯度)
使產生優勢的動作可能性更大：
```
L_actor = -E[log π(a|s) · A(s,a)]
```
如果 A > 0：增加該動作的機率
如果 A < 0：減少該動作的機率

### 2. 評論家損失 (價值函數 MSE)
使價值預測更準確（**MSE** = 均方誤差：對預測誤差取平方並求平均值——平方會對大錯誤比小錯誤給予更重的懲罰）：
```
L_critic = E[(V(s) - return)²]
```
就像訓練任何 **回歸 (regression)** 模型一樣（回歸 = 預測一個連續的數字，這裡是預期回報 V(s)）——最小化預測誤差。

### 3. 熵獎勵 (探索)
防止策略過快變得過於自信：
```
L_entropy = -H[π(·|s)] = E[log π(a|s)]
```
高熵 = 多樣化的動作選擇 = 探索
低熵 = 自信、狹窄的選擇 = 利用

**現實生活中的例子：** 熵獎勵就像一個老師說：「不要在每個多選題上都猜 A！嘗試不同的答案，這樣你才能學到什麼是有效的。」

```
總損失 = L_actor + 0.5 × L_critic - 0.01 × 熵
```

## LunarLander：更艱巨的挑戰

**LunarLander-v3** 是一個 Gymnasium（原 OpenAI Gym）環境——「v3」是版本號，表示該環境的第三次修訂。代理人控制一艘必須安全降落在月球指定平台上的小型航天器。它比 CartPole 難得多：
- 8 維狀態空間（位置、速度、角度、腿部接觸、燃料）
- 4 種離散動作（什麼都不做、啟動左引擎、啟動主引擎、啟動右引擎）
- 獎勵：成功著陸 +100，墜毀 -100，少量的燃料懲罰

訓練曲線顯示獎勵從極負值逐漸向正值提升。在 LunarLander 上運行 A2C 在著陸器學會基本穩定性之前需要大量的經驗。

## 關鍵方程

```
n 步回報:    G_t = r_t + γ·r_{t+1} + ... + γⁿ·V(s_{t+n})
優勢:        A_t = G_t - V(s_t)
演員更新:    θ_π ← θ_π - α ∇ L_actor
評論家更新:  θ_V ← θ_V - α ∇ L_critic
```

## 關鍵要點

| 概念 | 通俗解釋 |
|---------|---------------|
| **演員 (Actor)** | 策略——決定做什麼 |
| **評論家 (Critic)** | 價值函數——判斷情況有多好 |
| **優勢 (Advantage)** | 「這比預期的好嗎？」（實際 - 預期） |
| **N 步回報** | 在使用 V(s) 進行自舉之前，先向未來看 n 步 |
| **平行環境** | 多個環境以獲得去相關、多樣化的經驗 |
| **熵獎勵** | 鼓勵演員不斷嘗試新事物 |

## 下一步

A2C 很好，但它有一個主要弱點：有時它更新策略過於激進。一次糟糕的更新可能會摧毀之前更新中學到的所有好成果。

**PPO (Proximal Policy Optimization)** 透過一個聰明的「安全裁剪」來解決這個問題，防止任何單次更新對策略改變太大。

查看 `ppo_scratch.py` 瞭解 PPO 的實現！
