# PettingZoo 환경 탐색하기 🦓

## PettingZoo란 무엇인가요?

단일 에이전트 강화학습을 해보셨다면 아마 **Gymnasium** (OpenAI Gym의 후속작)을 사용해 보셨을 것입니다. 모든 환경은 똑같이 생겼습니다: `env.reset()`, `env.step(action) → obs, reward, done, info`. 즉, 세상에 대한 새로운 *관측*, 스칼라값인 *보상* 신호, 게임 끝을 알리는 *종료* 플래그, 그리고 디버깅용 정보를 담은 *인포* 딕셔너리로 구성되죠. 이러한 통일성이 강화학습 라이브러리를 작동하게 만드는 힘입니다.

**PettingZoo**는 정확히 같은 아이디어를 **다중 에이전트(multiple agents)**에 적용한 것입니다. 수많은 다중 에이전트 환경이 하나의 잘 정의된 API 뒤에 모여 있습니다.
- **클래식 토이 문제**: 알고리즘 테스트를 위한 가위바위보 같은 단순한 환경들.
- **협력형 그리드 월드**: 공유된 목표를 달성하기 위해 그리드를 누비는 에이전트들.
- **아타리 멀티플레이어**: 퐁(Pong)과 같은 클래식 경쟁 게임들.
- **MPE (Multi-Particle Environment)**: 복잡한 협력과 경쟁을 위한 연속 공간 물리 환경들.

하나의 PettingZoo 환경에서 작동하는 코드를 짤 수 있다면, 거의 수정 없이 다른 어떤 환경에도 바로 적용할 수 있습니다.

---

## 두 가지 API 스타일

다중 에이전트 환경은 두 에이전트가 동시에 행동할 수도, 차례대로 행동할 수도, 심지어 임의의 순서로 행동할 수도 있기 때문에 단일 에이전트보다 훨씬 복잡합니다. PettingZoo는 이를 위해 두 가지 병렬 API를 제공합니다.

### 1) AEC (Agent-Environment-Cycle)

한 번에 한 명의 에이전트만 행동합니다. 환경은 정해진 순서대로 에이전트들을 돌며 각자에게 다음을 제공합니다.
- **관측 (observation)**: 지금 이 순간 에이전트가 보는 것.
- **보상 (reward)**: 지난 라운드에서 수행된 **공동(joint)** 행동에 의한 대가 (즉, 나뿐만 아니라 *모든* 에이전트가 행동한 결과로 발생한 일입니다. 예를 들어 체스에서 보상은 내 수뿐만 아니라 상대방의 마지막 수 이후의 판 상태를 반영합니다).
- **종료 플래그 (termination flag)**: 에피소드가 *자연스럽게* 끝났을 때 (예: 체크메이트, 승리) `True`가 됩니다.
- **중단 플래그 (truncation flag)**: 자연스럽게 끝나기 전 시간 제한 등으로 에피소드가 *강제로* 중단되었을 때 `True`가 됩니다.

이는 체스, 바둑, 포커와 같은 **턴제 게임**에 자연스럽습니다.

```python
env.reset()
for agent in env.agent_iter():
    obs, reward, term, trunc, info = env.last()
    if term or trunc:
        env.step(None)
        continue
    action = my_policy(obs, agent)
    env.step(action)
```

### 2) Parallel (병렬)

모든 에이전트가 매 단계 동시에 관측하고 행동합니다. `step()` 함수는 행동들의 **딕셔너리**를 인자로 받고, 관측과 보상들의 딕셔너리를 반환합니다.

이는 MPE(모든 점 에이전트가 동시에 움직임)나 다중 에이전트 그리드 월드와 같은 **실시간 게임**에 자연스럽습니다.

```python
obs, info = env.reset()
while env.agents:
    actions = {a: my_policy(obs[a]) for a in env.agents}
    obs, rewards, terms, truncs, info = env.step(actions)
```

두 스타일은 **동형(isomorphic)**입니다. 즉, 구조적으로 동일하며 서로 변환 가능합니다. 어떤 AEC 환경이든 자동으로 Parallel 환경처럼 보이게 감쌀 수 있고, 그 반대도 가능합니다. PettingZoo는 이러한 변환 래퍼(wrapper)를 제공하므로 여러분은 한 가지 스타일에 맞춰 코드 한 벌만 작성하면 됩니다.

---

## 실생활 비유

- **AEC = 보드게임 밤.** "앨리스 차례, 이제 밥 차례, 그다음 캐럴 차례, 다시 앨리스." 다음 차례인 사람은 가장 최신의 판 상태를 봅니다.
- **Parallel = 멀티플레이어 비디오 게임.** 네 명의 플레이어 모두 동시에 버튼을 누르고 있고, 게임은 1초에 60번 세상을 업데이트합니다.
- **통일된 API가 중요한 이유.** 모든 멀티플레이어 게임마다 전용 조이스틱이 따로 필요하다고 상상해 보세요. 끔찍하겠죠? PettingZoo는 다중 에이전트 강화학습의 "범용 조이스틱"과 같습니다.

## 우리 코드의 작동 방식

우리는 밑바닥부터 **PettingZoo 스타일**의 환경인 **반복 협력 게임 (Iterated Coordination Game)**을 만듭니다. 두 에이전트가 반복적으로 채널 `0` 또는 `1`을 선택합니다.

- 같은 채널 선택 → 둘 다 +1 점
- 다른 채널 선택 → 둘 다 -1 점

각 에이전트가 받는 **관측**은 이전 라운드의 **공동 행동**입니다. 즉, 지난번에 두 에이전트가 무엇을 선택했는지가 하나의 정수로 묶여서 전달됩니다. 구체적으로, 각 에이전트의 지난 행동은 `{시작, 0, 1}` (3가지 상태) 중 하나이므로, 두 에이전트의 쌍은 `3 × 에이전트_1_상태 + 에이전트_2_상태`로 인코딩되어 9개의 정수(0 – 8)가 가능합니다. 정수 0은 아무런 행동도 취해지지 않은 "시작" 상태를 의미합니다. 한 에피소드는 25단계 동안 진행되므로 에이전트당 최대 점수는 +25, 최소 점수는 -25입니다. **무작위 플레이는 약 0점**을 기록합니다. 두 무작위 에이전트가 같은 것을 고를 확률 50%(+1)와 다를 확률 50%(-1)가 결합되어 단계당 기대 보상이 0이기 때문입니다.

우리는 다음을 수행합니다.

1. 무작위 실행을 통해 **AEC 인터페이스를 시연**합니다. `agent_iter()`가 차례인 에이전트를 알려주고, `last()`로 관측과 보상을 읽으며, `step()`으로 행동을 환경에 전달하는 기본 루프를 확인합니다.
2. **Parallel 인터페이스를 통해 두 개의 독립적인 Q-러너를 훈련**시킵니다. 각 에이전트는 **공동 행동 관측**(지난 라운드에 두 에이전트가 한 일을 인코딩한 정수)을 키로 하는 자신만의 Q-테이블을 가집니다. 이를 통해 "지난번에 둘 다 0을 골랐으니, 이번에도 0을 골라야지"라는 것을 배울 수 있습니다.
3. **실제 `pettingzoo` 라이브러리 임포트**를 시도하고, 내장된 환경 중 하나인 가위바위보(RPS)를 무작위 정책으로 실행해 봅니다. 라이브러리가 없다면 친절한 메시지와 함께 건너뜁니다.

### 예상 결과

| 단계 | 예상 결과 |
|-------|----------|
| 무작위 실행 (AEC)            | 평균 에피소드 리턴이 **0** 근처 — 무작위 에이전트들은 서로 독립적으로 채널을 고르므로 맞는 경우와 틀리는 경우가 비슷하게 발생합니다. |
| 독립적 Q-러너 (Parallel) — 초기 100회 | 약 **0** — 에이전트들이 탐험하는 동안에는 여전히 무작위에 가깝습니다. |
| 독립적 Q-러너 — 마지막 100회             | **+20에서 +25** 사이의 강력한 양수 — **협력이 발생**했습니다: 두 에이전트 모두 매 라운드 안정적으로 같은 채널을 고르는 법을 배웠습니다. |

`outputs/pettingzoo_coordination.png` 그래프는 개별 에피소드 리턴(회색)과 이동 평균 곡선(파란색)을 보여줍니다. 평균 곡선은 노이즈를 제거하여 에이전트들이 0점 근처의 무작위 플레이에서 +25점 근처의 안정적인 **협력** 상태로 나아가는 추세를 보여줍니다. 점선으로 된 초록색 선은 완벽한 협력을 의미하는 천장입니다.

라이브러리가 설치되어 있다면, 스크립트는 `pettingzoo.classic.rps_v2`를 실행하여 우리가 직접 만든 환경에서와 똑같은 방식으로 실제 라이브러리가 작동함을 증명합니다.

## 왜 커스텀 환경부터 만들어 보나요?

**API 그 자체가 교훈이기 때문입니다.** (구체적인 게임 규칙보다 다중 에이전트와 환경 사이의 상호작용을 어떻게 구조화하는지 이해하는 것이 더 중요합니다.) 다중 에이전트 강화학습에는 턴제, 실시간, 협력, 경쟁, 혼합형 등 다양한 형태가 있지만, 모두 AEC나 Parallel 패턴에 맞게 설계될 수 있습니다. 이 두 가지 루프만 구현해 보면, 다른 PettingZoo 환경들을 사용하는 것은 그저 생성자만 바꾸는 수준의 간단한 일이 됩니다. 훈련 코드는 그대로 유지되니까요.

이는 Gymnasium이 환경을 통일된 인터페이스 뒤의 블랙박스로 만들어 단일 에이전트 강화학습을 혁신했던 방식과 정확히 일치합니다.

## 독립적 Q-러닝의 장단점

협력 게임은 에이전트들이 보상 신호를 공유하므로 이해관계가 일치하여 *관대합니다*. 한 에이전트의 발전이 다른 에이전트에게도 도움이 되기 때문에 독립적인 학습자들도 즐겁게 문제를 해결할 수 있습니다.

하지만 가위바위보 같은 **대립형(adversarial)** 게임에서 독립적 Q-러닝은 영원히 요동칩니다. 한쪽이 적응하면 다른 쪽이 그에 맞춰 전략을 바꾸는 꼬리 물기가 끝없이 이어지기 때문입니다.
**부분 관측형(partially-observable)** 게임에서는 관측 정보가 상태의 일부분일 뿐이므로 아예 배우지 못할 수도 있습니다 (상대방이 무엇을 하는지 보지 못해 좋은 행동을 하고도 페널티를 받을 수 있습니다). PettingZoo는 이러한 실패 사례들을 직접 확인할 수 있도록 다양한 환경을 포함하고 있습니다.

## 기억해야 할 주요 용어

| 용어 | 의미 |
|------|---------|
| **PettingZoo**     | 다중 에이전트 강화학습의 Gymnasium — 표준화된 MARL 환경 라이브러리 |
| **AEC**            | Agent-Environment-Cycle: 단계당 한 에이전트가 행동함 (턴제) |
| **Parallel API**   | 모든 에이전트가 매 단계 동시에 행동함 |
| **MPE**            | PettingZoo에 포함된 인기 있는 협력/경쟁 테스트베드 (주로 물리 기반 과제를 수행하는 움직이는 점들) |
| **CTDE**           | Centralised Training, Decentralised Execution — 전체 정보를 보며 훈련하고, 실제 실행 시에는 자신의 시야만 사용하여 행동함 |
| **Independent Q-learning** | 다른 학습자의 존재를 무시하고 각자 일반적인 Q-러닝 알고리즘을 실행함 |

## 한 줄 요약

> **PettingZoo는 모든 다중 에이전트 환경을 같은 모양으로 만들어 줍니다. 따라서 오늘 짠 코드가 내일 전혀 다른 게임에서도 그대로 작동합니다.**

두 가지 API 스타일이 익숙해지면, MADDPG(연속 제어를 위한 중앙 집중식 비평가), QMIX(협력 팀을 위한 가치 믹싱), MAPPO(다중 에이전트 PPO) 등 어떤 현대적 MARL 알고리즘으로도 나아갈 수 있습니다. 환경 쪽 코드는 바꿀 필요가 없으니까요.
