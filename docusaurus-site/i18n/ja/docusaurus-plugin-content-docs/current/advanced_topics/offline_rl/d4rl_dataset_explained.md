# D4RL ベンチマーク・データセット 📦

## それは何ですか？

パンケーキをひっくり返すようにロボットに教えたいと想像してください。本物のコンロで1ヶ月間練習させるのは、時間がかかり、危険で、お金もかかります。しかし、あなたにはシェフがパンケーキをひっくり返している10年分のビデオ録画（上手なもの、下手なもの、ランダムなもの）があります。本物のフライパンに一度も触れさせることなく、**そのデータだけで**ロボットに教えることができるでしょうか？

それが**オフライン強化学習（offline reinforcement learning）**です。エージェントは過去の経験の固定されたデータセットから学習し、ライブの環境とは接触しません。最も難しいのは、エージェントが学習した内容を最後まで**試してみる**ことができない点です。

これを公平に研究するために、研究コミュニティは「標準的なデータセット」を必要としていました。それが **D4RL** (**D**atasets for **D**eep **D**ata-**D**riven **R**einforcement **L**earning) です。これは2020年にカリフォルニア大学バークレー校によってリリースされた、古典的な制御タスクのための事前録画された遷移データのコレクションです。すべての論文が同じバイトデータを使用してトレーニングするため、結果を比較できるようになっています。

## D4RLデータセットには何が入っているのか？

各タスクについて、D4RLは**4つの品質レベル**を提供しています：

| レベル | データの由来 | なぜ重要なのか |
|-------|---------------------------|----------------|
| **random**        | 行動を一様にランダムに選ぶ方策 | 最悪のケース：それでも何か有用なことを学べるか？ |
| **medium**        | 部分的にトレーニングされた方策（エキスパートの約半分のスコア） | 現実的：ほとんどのログデータは平凡である |
| **expert**        | ほぼ収束したエキスパート方策 | 最良のケース：ソースとなった方策に匹敵できるか？ |
| **medium-replay** | medium方策をトレーニングする際に使用された**リプレイバッファ全体** | 混合：初期の失敗と後の成功が混在している |

`medium` と `medium-replay` の違いは非常に重要です：
- **`medium`** は、単一の固定された「平均的」な方策に多くのゲームをプレイさせることで生成されます。すべてのデータが、この一定の平均的なスキルレベルを反映しています。
- **`medium-replay`** は歴史的なログです。ゼロから medium レベルまで**学習する過程**で収集されたすべての経験が含まれています。これは**悪い遷移と良い遷移**を混ぜ合わせています。これは、ロボットの最初の不器用な試行とその後の洗練された動作がすべて1つのバケツに入っている、現実世界のログと全く同じ姿です。

## オフラインデータセットの実生活の例

- **医療記録。** 何年分もの（患者の状態、治療、結果）のタプル。生きている人間にランダムな治療を行うことはできませんが、ログからより良い方策を学ぶことはできます。
- **カスタマーサービスのチャットログ。** 何百万もの（ユーザーのメッセージ、エージェントの返信、満足度）の記録。より多くのユーザーを煩わせることなく、より優れたアシスタントをトレーニングできます。
- **自動運転フリートのデータ。** テスラやウェイモのすべての車が走行データをアップロードします。そのフリート（車両群）は巨大な medium-replay データセットです。
- **推薦システム。** 去年のクリックログは固定されたデータセットです。同じ広告を同じユーザーに再度見せて試すことはできません。

これら4つのケースすべてにおいて、**環境に新しいサンプルを要求することはできません。** 手元にあるデータセットがすべてです。永遠に。

## コードの動作

実際のD4RLデータセットは、MuJoCo (Multi-Joint dynamics with Contact) というロコモーション（移動）タスク（HalfCheetah, Hopper, Walker2d, Ant など。これらは仮想ロボットが歩行や走行を学ぶ高度な3D物理シミュレーションです）で記録されています。MuJoCoはインストールが大変なため、以前のフェーズで学習した標準的な初心者用環境である **CartPole-v1 で同じ4レベルの構造**を再現します。教訓はそのまま転用できます。

スクリプト `d4rl_dataset.py` は以下のことを行います：

1. カートポールをクリア（報酬 475 以上）するまで **DQN** (Deep Q-Network) をトレーニングします。
2. その過程で**2つのチェックポイントをスナップショット**として保存します：
   - 「medium」 — 直近の報酬が 150 を超えた瞬間
   - 「expert」 — 直近の報酬が 475 を超えた瞬間
3. **medium方策のフルリプレイバッファ**をスナップショットとして保存します。これまでに見たすべての遷移データです。これが私たちの「medium-replay」データセットになります。
4. **3つの新しい方策をそれぞれ 10,000 遷移分実行**します：
   - `random`   — 一様ランダム
   - `medium`   — mediumチェックポイント + ε=0.10 のノイズ
   - `expert`   — expertチェックポイント + ε=0.02 のノイズ
5. **4つの `.npz` ファイル**（NumPyの圧縮配列形式）を `outputs/` に保存します。各ファイルには `obs / action / reward / next_obs / terminal` の配列が含まれています。

これら4つのファイルが、`cql.py` と `behavioral_cloning.py` への入力となります。

## 実行時に表示されるもの

コンソールに表示され、`outputs/d4rl_summary.txt` に保存されるプレーンテキストのサマリー：

```
データセット      |   N    |  平均報酬      |  最小  |  最大
------------------------------------------------------------
random          | 10000  |          ~22  |    ~9 |   ~80
medium          | 10000  |         ~180  |   ~50 |  ~500
expert          | 10000  |         ~490  |  ~400 |   500
medium-replay   | 10000  |          ~60  |    ~9 |  ~200
```

また、4つのデータセットがどのように重なり合っているかを示すヒストグラム (`outputs/d4rl_returns.png`) も生成されます。注目すべき主な特徴は以下の通りです：

- **Random** は 20 付近に集中します（ランダムな CartPole エピソードの平均的な長さ）。
- **Expert** は 500 の上限に集中します。
- **Medium** はその中間に位置し、分散が大きいです。
- **Medium-replay** は右側に長い裾（テール）を持っています。これは主に初期の失敗した試行（低い報酬）で構成されていますが、エージェントが学習するにつれて高い報酬へと伸びる裾を持っています。

## なぜデータセットが重要なのか

どのデータセットでオフラインアルゴリズムをトレーニングするかによって、可能なことの「限界（天井）」が決まります：

- **`expert` から学習する場合** — BC (Behavioral Cloning：単にデータをそのままコピーする手法) のような単純なアルゴリズムでもうまくいきます。なぜなら、すべてのデータが優れているからです。
- **`random` から学習する場合** — 稀にある「良い遷移」を**繋ぎ合わせる**（異なる試行からの良いアクションの短いシーケンスを組み合わせて成功への道筋を見つける）ことができるスマートなアルゴリズムが必要です。BCは完全に失敗します。
- **`medium-replay` から学習する場合** — 最も現実的で興味深いケースです。**CQL** (Conservative Q-Learning：見たことがないアクションに対して過信することを避ける手法) のような優れたアルゴリズムは、混合した信号から構造を抽出するため、時として**データの平均的な質を超える**ことができます。単純なアルゴリズム (BC) は平均へと回帰してしまいます。

## 覚えておくべきキーワード

| 単語 | 意味 |
|------|---------|
| **オフライン強化学習 (Offline RL)** | 固定されたデータセットからトレーニングする手法。環境との相互作用は許可されない |
| **行動方策 (Behaviour policy)** | データセットを**作成した**方策 |
| **データセットの質** | 行動方策がどれだけ優れていたか (random / medium / expert) |
| **リプレイバッファ (Replay buffer)** | トレーニング中に遭遇した遷移データの全履歴 |
| **分布シフト (Distribution shift)** | データセット内のアクションと、トレーニングされた方策が取りたいアクションの間のギャップ。データセットには新しい方策が記録されていないことを試したときに何が起こるかが示されていないため、アルゴリズムの価値推定が危険なほど間違ってしまう可能性がある。 |
