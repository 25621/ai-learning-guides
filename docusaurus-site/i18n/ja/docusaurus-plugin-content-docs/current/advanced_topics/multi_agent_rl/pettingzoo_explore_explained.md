# PettingZoo環境の探索 🦓

## PettingZooとは何か？

シングルエージェントの強化学習を行ったことがある人なら、おそらく**Gymnasium**（OpenAI Gymの後継）を使ったことがあるでしょう。すべての環境は、`env.reset()`、`env.step(action) → obs, reward, done, info` という同じ形式をしています。つまり、世界の新しい「観測（observation）」、スカラー値の「報酬（reward）」、ゲームオーバーを知らせる「終了（done）」フラグ、そしてデバッグ用の「追加情報（info）」辞書です。この統一性こそが、強化学習ライブラリを機能させている要因です。

**PettingZoo**は、これと全く同じアイデアを**マルチエージェント**に適用したものです。PettingZooはマルチエージェント環境の「動物園（zoo）」であり、すべてが以下の定義されたAPIの背後にあります。

- **古典的なトイ・プロブレム**：基本的なアルゴリズムをテストするための「じゃんけん」のようなシンプルな環境。
- **協調的なグリッドワールド**：共通の目標を達成するためにグリッド内を移動するエージェント。
- **Atariマルチプレイヤー**：「Pong」のような古典的な対戦ゲーム。
- **MPE (Multi-Particle Environment)**：複雑な協調や競争のための、連続空間における物理環境。

一つのPettingZoo環境で動作するコードを書くことができれば、他のどの環境にもほとんど変更なしで適用できます。

## 2つのAPIスタイル

マルチエージェントの設定は、2つのエージェントが同時に行動したり、順番に行動したり、さらには不規則な順序で行動したりするため、シングルエージェントよりも複雑です。PettingZooは、これを2つの並行するAPIで解決しています。

### 1) AEC (Agent-Environment-Cycle)

一度に一人のエージェントが行動します。環境はある順序でエージェントをループし、各エージェントは以下を受け取ります：
- **観測（observation）**：そのエージェントが「今」見ているもの。
- **報酬（reward）**：直前のラウンドにおける「共同行動（joint action）」によって得られた利益（つまり、自分だけでなく全エージェントの行動の結果として起きたこと。例えばチェスでは、報酬は自分の手だけでなく、相手の最後の手の後の盤面を反映します）。
- **終了（termination）フラグ**：エピソードが「自然に」終了した（例：チェックメイト、誰かの勝利）場合に `True`。
- **打ち切り（truncation）フラグ**：自然な終了に達する前に、制限時間などによってエピソードが「途中で打ち切られた」場合に `True`。

これは、チェス、囲碁、ポーカーのような**ターン制ゲーム**に適しています。

```python
env.reset()
for agent in env.agent_iter():
    obs, reward, term, trunc, info = env.last()
    if term or trunc:
        env.step(None)
        continue
    action = my_policy(obs, agent)
    env.step(action)
```

### 2) Parallel

すべてのエージェントが毎ステップ同時に観測し、行動します。`step()` は行動の**辞書**を受け取り、観測と報酬の辞書を返します。

これは、MPE（すべてのドット状のエージェントが同時に動く環境）やマルチエージェント・グリッドワールドのような**リアルタイムゲーム**に適しています。

```python
obs, info = env.reset()
while env.agents:
    actions = {a: my_policy(obs[a]) for a in env.agents}
    obs, rewards, terms, truncs, info = env.step(actions)
```

これら2つのスタイルは**同型（isomorphic）**、つまり構造的に等価であり、相互に変換可能です。AEC環境をParallel環境のように見せる自動ラップ機能（およびその逆）が備わっています。PettingZooには変換用のラッパーが含まれているため、どちらか一方のスタイルでコードを書くだけで済みます。

## 現実世界の例え

- **AEC = ボードゲームの夜**。「アリスの番。次はボブ。次はキャロル。アリスに戻る」。次に動く人は、最新の盤面を見ることができます。
- **Parallel = マルチプレイヤー・ビデオゲーム**。4人のプレイヤー全員が同時にボタンを押しており、ゲームは1秒間に60回世界を更新します。
- **なぜ統一されたAPIが重要なのか**：もしマルチプレイヤー・ビデオゲームごとに専用のジョイスティックが必要だとしたら大変です。PettingZooは、マルチエージェント強化学習における「万能ジョイスティック」なのです。

## コードが行っていること

ゼロから**PettingZooスタイル**の環境を構築します。題材は「**反復協調ゲーム（Iterated Coordination Game）**」です。2人のエージェントが繰り返しチャンネル `0` または `1` を選びます。

- 同じ選択をした場合 → 両者が +1
- 異なる選択をした場合 → 両者が -1

各エージェントが受け取る**観測**は、直前の「共同行動」です。つまり、前のラウンドで両者が何を選んだかを一つの整数にまとめたものです。具体的には、各エージェントの最後の行動は `{開始, 0, 1}` の3状態なので、ペアの状態は `3 × エージェント1の状態 + エージェント2の状態` となり、9つの整数（0〜8）でエンコードされます。整数0は「開始」状態で、まだ行動が行われていない（エピソードの最初）ことを示します。エピソードは25ステップ続くため、1エージェントあたりの最大合計リターンは +25、最小は -25 です。**ランダムなプレイではスコアは約0になります**。なぜなら、各ステップで2人の独立したランダムなエージェントが等確率で0または1を選ぶため、一致する確率が50%（+1）、異なる確率が50%（-1）となり、1ステップあたりの期待報酬は `0.5 × (+1) + 0.5 × (−1) = 0` となるからです。25ステップ合計の期待リターンも同様に0となります。

コードでは以下のことを行います：

1. **ランダムなロールアウトによるAECインターフェースのデモンストレーション**：基本的なAECループを確認します。`agent_iter()` は手番のエージェントを返し、`last()` はそのエージェントの現在の観測と累積報酬を読み取り、`step()` は選択された行動を環境に返します。
2. **Parallelインターフェースを通じた2つの独立したQ学習者のトレーニング**：各エージェントは、直前の共同行動（両者が何をしたかを表す整数）をキーとする自身のQテーブルを持ちます。これにより、「前回二人とも0を選んだなら、今回も0を選ぶべきだ」といった学習が可能になります。
3. **本物の `pettingzoo` ライブラリのインポートの試行**：ライブラリがインストールされている場合、組み込み環境（じゃんけん）の一つをランダムポリシーで実行します。インストールされていない場合は、メッセージを表示してこのステップをスキップします。

### 確認できること

| ステージ | 期待される結果 |
| :--- | :--- |
| ランダム・ロールアウト (AEC) | エピソード平均リターンが **0** に近い。ランダムなエージェントは独立にチャンネルを選ぶため、一致と不一致がほぼ同数になる。 |
| 独立Q学習 (Parallel) — 最初の100エピソード | 約 **0**。探索中のため、まだほとんどランダム。 |
| 独立Q学習 — 最後の100エピソード | 非常に高いプラス（**+20 〜 +25**）。**「協調」が芽生えました**。両方のエージェントが、毎ラウンド同じチャンネルを確実に選ぶことを学習しました。 |

`outputs/pettingzoo_coordination.png` のプロットには、個々のエピソードのリターン（灰色）と、移動平均の曲線（青色）が表示されます。平均曲線によってノイズが取り除かれ、傾向が見て取れます。エージェントが約0付近の無秩序なランダムプレイから、+25付近の安定した**協調**へと移行していく様子がわかります。緑色の点線は、完璧な協調の天井（上限）を示しています。

`pettingzoo` がインストールされている場合、スクリプトは `pettingzoo.classic.rps_v2` も実行し、自作の環境と同じように本物のライブラリに対してもコードが機能することを証明します。このセクションを有効にするには、以下を実行してください：

```bash
source ../../venv/bin/activate
pip install "pettingzoo[classic]"
```

## なぜ最初にカスタム環境を自作するのか？

それは、**「APIそのものが教訓（レッスン）」だから**です。（複数のエージェントと環境の間の対話をどのように構造化するかを理解することは、特定のゲームルールを理解することよりも重要です。）マルチエージェント強化学習には多くの種類（ターン制、リアルタイム、協調、競争、混合）がありますが、それらすべてがAEC / Parallelのパターンに当てはまります。これら2つのループを一度実装してしまえば、他のどのPettingZoo環境も、コンストラクタを入れ替えるだけで済み、トレーニング用のコードはそのまま使えます。

これは、Gymnasiumが環境を統一されたインターフェースの背後にあるブラックボックスにすることで、シングルエージェント強化学習に変革をもたらしたのと全く同じ仕組みです。

## 独立Q学習の長所と短所

協調ゲームは「寛容」です。エージェント間で報酬の符号が共有されているため、利害が一致します。一方のエージェントの改善がもう一方の助けにもなるため、独立した学習者でもこれらを円満に解決できます。

一方、**対戦型**ゲーム（じゃんけんなど）では、独立Q学習は永遠に振動し続けます（一方が適応すると、もう一方がそれを打ち負かすように戦略を変え、いたちごっこが続くため）。**部分観測**ゲームでは、観測が状態の一部しか表していないため、全く学習できないことがあります（相手が何をしているか見えないため、良い行動をとったのに罰せられることがあるからです）。PettingZooにはこれら両方の環境が含まれているため、こうした失敗モードを自分自身で確認することができます。

## 覚えておくべきキーワード

| 単語 | 意味 |
| :--- | :--- |
| **PettingZoo** | マルチエージェント強化学習版のGymnasium。標準化されたMARL環境のライブラリ。 |
| **AEC** | Agent-Environment-Cycle。ステップごとに一人のエージェントが行動する（ターン制）。 |
| **Parallel API** | 各ステップで全エージェントが同時に行動するAPI。 |
| **MPE** | Multi-Particle Environment。PettingZooに付属する人気の協調/対戦テストベッド（物理ベースのタスクを行うドット状のエージェントがよく登場する）。 |
| **CTDE** | Centralised Training, Decentralised Execution。中央集中型学習・分散型実行。学習時はグローバルな情報（全状態）にアクセスできるが、実行時はローカルな観測（各自の限られた視界）のみで動くこと。 |
| **独立Q学習 (Independent Q-learning)** | 他の学習者の存在を無視して、標準的な（修正なしの）Q学習を各エージェントが個別に実行すること。 |

## 一行まとめ

> **PettingZooは、あらゆるマルチエージェント環境を同じ形状（API）に収めます。そのため、今日書いたコードは、明日全く別のゲームでもそのまま機能します。**

これら2つのAPIスタイルが身につけば、MADDPG（連続制御エージェントのための中央集中型評論家）、QMIX（協調チームのための価値混合）、MAPPO（マルチエージェントPPO）、あるいはその他の現代的なMARLアルゴリズムへとステップアップすることができます。その際、環境側のコードを変更する必要は一切ありません。
