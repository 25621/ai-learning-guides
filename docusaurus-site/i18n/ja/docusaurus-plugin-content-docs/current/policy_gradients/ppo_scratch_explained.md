# PPO：安全で着実な方策更新

## A2Cの問題点

指の上でほうきのバランスを取る練習をしているところを想像してください。数週間の練習の後、あなたは30秒間も立てていられるようになりました！

ここでコーチがアドバイスをくれます。「手首をもう少しだけ左に傾けてごらん。」

**良いアドバイス → 慎重な変更 → まだ30秒間バランスを保てる ✓**

しかし、もしコーチが過剰に反応して「今すぐ思いっきり左に傾けろ！」と言ったらどうなるでしょう？
あなたは修正しすぎます → ほうきが倒れます → 数週間の進歩が台無しになります。

これがA2Cの問題です：**大きな勾配の更新が、せっかくの良い方策を破壊してしまう可能性がある**のです。

**PPO (Proximal Policy Optimization)** は、これを防ぐための安全システムです。

---

## コアとなるアイデア：うまくいっていた状態の近くに留まる

PPOの主要な制約：

> **「1回の更新で方策を大きく変えすぎてはいけない。」**

更新前には「古い」方策 π_old があります。
更新後には「新しい」方策 π_new があります。

PPOは、**確率比（probability ratio）**を使って方策がどれだけ変わったかを測定します：

```
r(θ) = π_new(a|s) / π_old(a|s)
```

- r = 1.0: 方策は変わっていない
- r = 1.5: 新しい方策はその行動をとる確率が50%高くなった
- r = 0.5: 新しい方策はその行動をとる確率が50%低くなった

**実生活の例：** あなたはレシピを調整しているシェフです。
- r = 1.0: 以前と同じ量の塩
- r = 2.0: 塩を2倍にする — 極端すぎます！
- r = 0.9: 塩を10%減らす — 小さくて安全な変更

---

## クリッピング・トリック (The Clipping Trick)

PPOは比率を [1-ε, 1+ε] の範囲内に収まるようにクリップ（制限）します（通常 ε = 0.2）：

```
L_CLIP = E[min(r(θ) · A,  clip(r(θ), 1-ε, 1+ε) · A)]
```

これを分解して考えてみましょう：

**ケース 1: 行動が良かった場合 (A > 0)**

その行動をもっと増やしたい（r > 1）と考えます。しかし、増加させる量に上限を設けます：
```
もし r > 1.2 なら： 1.2 にクリップし、それ以上増やす動機を与えない
```
これにより、一方向に極端に振れすぎるのを防ぎます。

**ケース 2: 行動が悪かった場合 (A < 0)**

その行動を減らしたい（r < 1）と考えます。ここでも上限を設けます：
```
もし r < 0.8 なら： 0.8 にクリップし、それ以上減らす罰を与えない
```

**視覚的イメージ：**
```
ε = 0.2 なので、安全な比率の窓は 0.8 から 1.2 です。

良い行動 (A > 0): 行動確率を上げますが、1.2 を超えたら報酬を停止します。
比率 r:       0.6      0.8      1.0      1.2      1.4
インセンティブ:  ↑        ↑        ↑        ↑        -
意味:        低すぎ     ok      古い      最大    クリップ

悪い行動 (A < 0): 行動確率を下げますが、0.8 を下回ったら報酬を停止します。
比率 r:       0.6      0.8      1.0      1.2      1.4
インセンティブ:  -        ↓        ↓        ↓        ↓
意味:       クリップ   最大     古い      ok      高すぎ
```

`-` は平らなクリップされた領域を示します。その領域では、確率比をさらに極端にしても目的関数は改善されないため、PPOにはそれ以上無理に押し進める動機がなくなります。

**実生活の例：** 車のスピードリミッター。加速はできますが、120km/hに達するとリミッターが作動し、それ以上速くならないようにします。動きを止めることなく、安全を確保します。

## なぜこれが壊滅的な更新を防ぐのか

**壊滅的な更新（catastrophic update）**とは、一度の大きな方策変更によって、エージェントがそれまでに学んだすべてが完全に破壊されてしまうことです。数時間のトレーニングが、たった一つの勾配ステップで消え去ってしまいます。

クリッピングがない場合：一度の大きな勾配ステップが方策を劇的に変えてしまう可能性があります。
クリッピングがある場合：比率が [1-ε, 1+ε] の外側では勾配がゼロになるため、方策は1ステップごとに少しずつしか動けません。

**実生活の例：** 優秀な外科医は、大きくざっくり切るのではなく、小さく正確に切開します。PPOは、強化学習オプティマイザにおける「慎重な外科医」なのです。

## GAE：より賢いアドバンテージ推定

PPOは **GAE (Generalized Advantage Estimation)** を使用してアドバンテージを計算します：

```
δ_t = r_t + γ · V(s_{t+1}) - V(s_t)          (TD 誤差)
A_t = δ_t + γλ · δ_{t+1} + (γλ)² · δ_{t+2} + ...
```

GAEにはパラメータ λ（ラムダ）があります：
- λ = 0: 1ステップのTD誤差のみを使用（分散は低いが、バイアスが高い）
- λ = 1: 完全なモンテカルロ・リターンを使用（分散は高いが、バイアスが低い）
- λ = 0.95: 両者の優れたバランス！

**実生活の例：** ロードトリップの計画。
- λ=0: 次の5マイルだけを見る（安全だが、先の近道を逃すかもしれない）
- λ=1: 500マイルの全行程を考慮する（情報は多いが、不確実性が高い）
- λ=0.95: 遠くを見通しつつ、近くの道をより重視する ← 最高のバランス！

## マルチエポック：データを効率的に再利用する

REINFORCEでは、集めた経験（ロールアウト）を1回の更新後に捨ててしまいます。

PPOは、各バッチを **K エポック**（通常、同じデータに対して4〜10回のパス）にわたって再利用します：

```
512ステップ × 4環境 = 2048個の遷移データを収集
エポック 1: 32個のミニバッチに分けて、それぞれを更新
エポック 2: シャッフルして、再び32個のミニバッチで更新
エポック 3: ...
エポック 4: ...
```

**「ミニバッチ（minibatch）」とは？** 2048個のデータすべてを一度に更新に使うと速度が遅く、メモリも消費します。一方で、データを1個ずつ使うとノイズが大きすぎます。**ミニバッチ**はその中間の小さな塊です。ここでは 2048 ÷ 32 = **1ミニバッチあたり64個のデータ**になります。1ミニバッチごとに1回の勾配ステップを計算するため、各エポックで1回の巨大な更新の代わりに、32回の小さくて安定した更新が行われます。（これはディープラーニング全般で使用されるミニバッチと同じ考え方です。詳細は [ミニバッチ勾配降下法](https://ja.wikipedia.org/wiki/%E7%A2%BA%E7%8E%87%E7%9A%84%E5%8B%BE%E9%85%8D%E9%99%8D%E4%B8%8B%E6%B3%95#%E3%83%9F%E3%83%8B%E3%83%90%E3%83%83%E3%83%81%E6%B3%95) を参照してください。）

クリッピングがあるおかげで、これらの複数回のパスでもオーバーシュートが起きません。クリッピングがなければ、複数回のエポックは方策を遠くに押しやりすぎて破壊してしまうでしょう！

**実生活の例：** 学生が30問の練習問題を解いています。
- REINFORCE：各問題を1回解き、少しだけ学んでから問題を捨てる。
- PPO：各問題を4回解く（毎回違う角度から見直す）。ただし、一度に考え方を変えすぎないように制限（クリップ）をかけ、間違ったパターンを丸暗記しないようにする。

## 完全な PPO 損失関数

```
L = L_CLIP - c₁ · L_entropy + c₂ · L_critic

L_CLIP    = クリックされた方策勾配
L_entropy = エントロピーボーナス (探索を促す)
L_critic  = V(s) と実際のリターンの間の MSE
```

一般的な係数：c₁ = 0.01（エントロピー）、c₂ = 0.5（価値関数）

**詳しく見ておくべき2つの項：**

- **方策勾配 (Policy gradient)** — 損失の「Actor」側。勾配信号を使用して、方策をより高いアドバンテージを持つ行動へと押し上げ、低いアドバンテージの行動から遠ざけます。これはREINFORCEで導入されたのと同じコアアイデアです。直感的な理解については [REINFORCEの解説](./reinforce_cartpole_explained.md#the-old-way-vs-the-new-way) を参照してください。PPOはそこにクリッピングの包み紙を追加しただけです。
- **MSE (Mean Squared Error：平均二乗誤差)** — 損失の「Critic」側。Critic V(s) は、ある状態からの期待リターンを予測します。その予測を実際のリターンと比較し、差を2乗します：`MSE = mean((V(s) - return)²) `。2乗することで、小さなミスよりも大きなミスを重く罰し、トレーニングのための滑らかで微分可能な信号を提供します。（標準的な回帰損失です。詳細は [平均二乗誤差](https://ja.wikipedia.org/wiki/%E5%B9%B3%E5%9D%87%E4%BA%8C%E4%B9%97%E8%AA%A4%E5%B7%AE) を参照してください。）

## 結果

```
更新  200回 | 平均報酬: 〜120
更新  400回 | 平均報酬: 〜280
更新  800回 | 平均報酬: 〜280-300
```

CartPole上のPPOは着実な改善を示しますが、報酬280〜300あたりでプラトー（停滞）に達する傾向があります。
（**プラトー**とは、学習曲線が平坦になり、トレーニングを続けても報酬が改善しなくなる状態を指します。方策が局所的に良い戦略を見つけたものの、それ以上の進歩が止まってしまった状態です。）
これは実は予想通りです。PPOは、より困難でエピソードが長い環境向けに設計されているからです。

興味深い観察結果：**REINFORCEの方がCartPoleを速く解いた！**（平均500 vs 平均300）

なぜでしょうか？ CartPoleのエピソードは短いため（500ステップ以下）、REINFORCEが計算する正確なリターンが非常に正確だからです。PPOのブートストラップによる推定は、不要な複雑さを加えてしまいます。PPOが真に本領を発揮するのは、エピソードが完結するのを待つのが現実的ではない環境（BipedalWalkerなど）です。

**「BipedalWalker」とは？** BipedalWalker（具体的には [Gymnasium](https://gymnasium.farama.org/environments/box2d/bipedal_walker/) の `BipedalWalker-v3`）は、強化学習の古典的なベンチマーク環境です。2本脚のロボットが、転倒せずに凹凸のある地面を前方に歩くことを学習しなければなりません。CartPoleの2つの離散アクション（左/右）とは異なり、BipedalWalkerは**連続値**アクション（各脚の関節、合計4つのトルク値、各 [-1, 1] の実数）を持ちます。エピソードは数千ステップに及ぶこともあり、まさにPPOのデータ効率と安定性が報われる領域です。

---

## 重要な数式

```
比率:      r_t(θ) = π_θ(a_t|s_t) / π_θ_old(a_t|s_t)
クリップ損失: L_CLIP = E[min(r_t A_t, clip(r_t, 1-ε, 1+ε) · A_t)]
GAE:        A_t = Σ_{l=0}^{∞} (γλ)^l · δ_{t+l}
```

---

## 重要なポイント

| コンセプト | 平易な説明 |
|---------|---------------|
| **比率 r(θ)** | その行動において、方策がどれだけ変わったか |
| **クリップ ε** | 安全境界線。これ以上方策を大きく変えない |
| **GAE** | 数ステップ先まで見ることでアドバンテージを賢く推定する手法 |
| **データ効率** | 各ロールアウトは複数の並列環境（相関のない安定した経験）から収集され、Kエポックのミニバッチ更新で再利用されます。クリッピングがこれらの繰り返しパスを安全に保ちます。 |

## 次は？

これまでの環境はすべて、**離散的（discrete）**なアクション（左または右に押す）でした。

現実のロボットは、「正確に 0.73 ニュートンの力を加える」といった**連続的（continuous）**なアクションを制御する必要があります。

`ppo_continuous.py` では、**ガウス方策**を使用してPPOを連続値アクションに拡張し、より難易度の高い BipedalWalker-v3 環境でテストします！
