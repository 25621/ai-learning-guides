# 連続値制御のためのPPO：BipedalWalkerを歩かせる

## 離散的アクション vs. 連続的アクション

これまでのところ、解決してきたすべての環境は**離散的（discrete）**なアクションを持っていました：
- CartPole：左に押すか、右に押すか（2つの選択肢）
- LunarLander：何もしない / 左 / メイン / 右を噴射（4つの選択肢）

しかし、現実世界のロボットには**連続的（continuous）**なアクションが必要です：
- ヒューマノイドロボット：「各関節をどれくらいの力で押すか」（-1から+1までの任意の値）
- 車：「ハンドルを正確にどれくらい回すか」（-30°から+30°までの任意の角度）
- ロボットアーム：「この方向に正確に2.3ニュートンの力を加える」

**実生活の例：** キーボードでタイピングすること = 離散的（A、B、C...を押す）。鉛筆で書くこと = 連続的（手を右に2.3cm動かし、40gの圧力をかける...）。

## 連続的アクションのためのガウス方策

連続的アクションの場合、カテゴリカル分布（N個のカテゴリから選択する）の代わりに、**正規分布（ガウス分布）**を使用します：

```
アクション ~ 正規分布(μ, σ)
```

ここで：
- **μ (ミュー、平均)**：分布の中心。ネットワークが「狙う」アクションの値。
- **σ (シグマ、標準偏差)**：広がり。どれだけのランダム性 / 探索を加えるか。

```
        確率
         │
    0.4 ─┤      ██████
         │    ████████████
    0.2 ─┤  ██████████████████
         │████████████████████████
         └──────────────────────── アクションの値
       -1  -0.5   0   0.5   1
                  ↑
                平均 μ
```

**実生活の例：** 熟練した射手は的の中心（μ）を狙います。矢のすべてが全く同じ場所に当たるわけではなく、多少のバラつき（σ）があります。練習を積むにつれて、中心を狙ったまま精度が高まります（σが減少します）。

## 私たちのガウス Actor-Critic ネットワーク

```
状態 (24個の数字) → [256 ユニット] → [256 ユニット] →
    ├── Actor: 4つの平均値  (μ₁, μ₂, μ₃, μ₄)
    │          + 4つの log_std パラメータ (すべての状態で共有！)
    └── Critic: 1つの価値 (V(s))
```

`log_std`（**標準偏差**の対数。広がりや不確実性の尺度）は、状態に依存しない**学習可能なパラメータ**です。これにより、トレーニング中に探索率を変化させつつ、シンプルさを保つことができます。

**なぜ std ではなく log_std なのか？** 標準偏差は正の値である必要があります。`log_std`を使用すると、ネットワークは任意の実数（正または負）を出力でき、その後に `exp(log_std)`（対数の逆関数である指数関数）を適用することで、確実に正の std を得ることができます。これにより、std が負やゼロになるのを防ぎます。

## 連続的アクションの対数確率の計算

離散的アクションの場合：`log_prob = log(P(action=LEFT))`

連続的アクションの場合、**正規分布**は平均を中心とした滑らかな鐘形の曲線を描きます。連続値の数学では、単一の正確な値が持つ確率はゼロであるため、その値における曲線の高さを使用します。これは **pdf**（確率密度関数：probability density function）と呼ばれます。
```
log_prob = Σᵢ log[正規分布(μᵢ, σᵢ).pdf(aᵢ)]
```

`log` は自然対数を意味します。非常に小さな密度の値を、ニューラルネットワークが最適化しやすい安定した数値に変換します。完全なアクションは4つの数字からなる1つのベクトルであるため、すべてのアクション次元（BipedalWalkerの場合は4つ）にわたって合計します。

**実生活の例：** 明日の気温が正確に 5.732...°C になる確率はどれくらいでしょうか？連続的な天候の場合、正規分布の曲線を見て、その正確なポイントでの高さを確認します。より起こりそうな気温（平均に近い）ほど、確率は高くなります。

## BipedalWalker：歩行の挑戦

BipedalWalker-v3 は、転倒せずに歩くことを学ばなければならない2Dロボットです：

```
          O (頭)
         /│\
        / │ \
       /  │  \
      L   │   R   ← 2本の脚、それぞれに膝関節
     / \  │  / \
    ●   ● │ ●   ●  ← 4つのモーター (各脚の股関節/膝関節)
```

**状態空間 (24個の数字):**
- 船体（Hull）：角度、角速度、水平速度、垂直速度 (4個)
- 関節（Joints）：4つのモーター（2つの股関節、2つの膝関節）がそれぞれ角度と速度を提供し、さらに2つの接地センサー（各脚に1つ） (10個)
- 10個のLIDARレンジセンサー（前方の地面を見る距離の読み取り値） (10個)

**アクション空間 (4つの連続値、各 [-1, 1]):**
アクション値は、正確に4つの関節に加えられる**トルク**（モーターによって加えられる回転力）を制御します（船体に直接アクションが加えられることはありません）：
- 脚1 股関節トルク、脚1 膝関節トルク、脚2 股関節トルク、脚2 膝関節トルク

**報酬:**
- ゴール（右側）に到達すると +300
- 転倒（体が地面に触れる）すると -100
- 前進するステップごとに小さな報酬
- エンジン使用ごとに小さなペナルティ（効率性のための報酬）

**クリア条件:** 100エピソードの平均報酬 > 300

## 離散的PPOとの主な違い

以下の点を除いて、すべて同じです：

| | 離散的PPO | 連続的PPO |
|---|---|---|
| **方策 (Policy)** | Categorical(logits) | 正規分布(μ, σ) |
| **サンプリング** | アクション = {0,1,...,N} からサンプリング | アクション = μ + σ × ノイズ |
| **log_prob** | log P(action=k) | Σ log 正規分布(μᵢ, σᵢ).pdf(aᵢ) |
| **クランプ** | 不要 | アクションを [-1, 1] にクランプする |

**Logits** は、離散的アクションに対する生の正規化されていないスコアです。カテゴリカル方策は、それらを **softmax**（任意の数値のセットを、すべての値が正で合計が1になる有効な確率分布に押し込める関数）によって確率に変換します。例えば、ロジット [2.0, 1.0, 0.5] は確率 [0.59, 0.24, 0.17] になります。連続的PPOでは、アクションが固定されたメニューから選ばれるわけではないため、アクション自体に softmax は使用**しません**。代わりに、方策は正規分布の平均と標準偏差を出力し、そこから実数値のトルクをサンプリングします。

**クランプ（Clamp）**とは、値を有効な範囲内に強制的に収めることを意味します。コードでは `action.clamp(-1, 1)` を使用しているため、環境が許容範囲外のモーターコマンドを受け取ることはありません。

PPOにおける**クリップ（Clip）**は別の意味を持ちます。PPOは、[PPOクリッピングのセクション](./ppo_scratch_explained.md#クリッピングトリック-the-clipping-trick)で説明されているように、損失関数内の確率比をクリップします。アクションのクランプは環境インターフェースを保護し、PPOのクリッピングは方策の更新を保護します。

## ゼロからの歩行：エージェントが学ぶこと

**トレーニング初期（マイナスの報酬）：** ロボットはランダムに暴れ、すぐに転倒します。すべてのエピソードが数秒以内にクラッシュで終わります。

**トレーニング中期：** ロボットは、脚を交互に動かすことで前進できることを発見します。ぎこちない小さなステップを踏み始め、報酬のマイナスが少なくなります。

**トレーニング後期：** スムーズで効率的な歩行の**歩容（gait）**が現れます。歩容とは、左右のステップを交互に繰り返すような繰り返しの運動パターンのことです。ロボットはLIDARセンサーを活用してリアルタイムでステップを調整し、凹凸のある地形にもダイナミックに適応します。

**実生活の例：** 赤ちゃんが歩き方を学ぶとき：
1. すぐに転ぶ（マイナスの報酬）
2. 一歩踏み出して転ぶ（少しだけマイナスが減る）
3. 数歩歩く（小さなプラスの報酬）
4. 部屋を横切って歩く（大きなプラスの報酬！）

## なぜ BipedalWalker には PPO が必要なのか (REINFORCE ではなく)

- **BipedalWalker のエピソード**は最大1600ステップに達することがあります（CartPoleよりもはるかに長いです！）。
- **報酬が疎（sparse）**です。前進による報酬は、1ステップあたりではごくわずかです。
- **REINFORCE の場合**、有用な信号を得るために何千もの完全なエピソードが必要になります。

PPO の [GAE (Generalized Advantage Estimation)](./ppo_scratch_explained.md#gaeより賢いアドバンテージ推定) を伴う n ステップ更新により、ロボットは不完全なエピソードからも学習できます：
> 「50ステップで転んでしまったけれど、その50ステップの間はある程度前進できていた。
> エピソードが終わるのを待つのではなく、50ステップのリターン推定値を使おう。」

## 結果

500回の更新（約100万環境ステップ）後：
- ロボットはランダムな暴れから、ある程度の前進へと目に見える進歩を遂げます。
- 学習曲線に一貫した改善が見られます。
- 報酬 > 300 への完全な収束には、さらなるトレーニング（500万〜1000万ステップ）が必要です。

学習曲線は、連続値制御に特有の「S字カーブ」を示します：
1. 初期段階の遅い進歩（学習の安定化）
2. 急激な向上（歩容の発見）
3. 緩やかな洗練（歩容の最適化）

## 重要なポイント

| コンセプト | 平易な説明 |
|---------|---------------|
| **ガウス方策** | メニューから選ぶ代わりに、ある範囲の値にダーツを投げる |
| **μ (平均)** | 方策が「狙う」場所 |
| **σ (標準偏差)** | 方策が使用するランダム性 / 探索の量 |
| **学習可能なパラメータとしての log_std** | 他のネットワークの重みと同様に、勾配ベースの最適化（報酬に対する勾配*上昇*、またはPPO損失に対する勾配*降下*）によって更新されるグローバルな探索率 |
| **連続値制御** | 実数値の出力（トルク、力、角度）を制御すること |

## 次は？

PPOには多くの**ハイパーパラメータ**があります。これは、トレーニング開始前に選択する設定のことです（自動的に学習されるネットワークの重みなどの「パラメータ」とは異なります）。例として、`clip_eps`、学習率、エポック数、バッチサイズなどがあります。

PPOはこれらの選択に対してどれほど敏感なのでしょうか？ `ppo_hyperparams.py` では、各ハイパーパラメータを系統的に変化させる実験を行い、学習速度と安定性への影響を示します。
