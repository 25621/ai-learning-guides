# REINFORCE：ロボットに、より良い選択を教える

## 私たちは何をしようとしているのか？

あなたがビデオゲームをしているロボットを持っていると想像してください。毎秒、ロボットは選択しなければなりません：
**「ボタンを押すべきか、押さないべきか？」**

Q学習のようにすべての状況をテーブルに記憶する代わりに、ロボットに「この状況では、この行動をせよ」と直接指示する一連のルール、つまり**レシピ**を学ばせたいと考えています。

このレシピは**方策（ポリシー、π）**と呼ばれます。強化学習において、πは「行動を選択するためのルール」を意味します。

---

## 従来の方法 vs. 新しい方法 {#the-old-way-vs-the-new-way}

**従来の方法 (Q-learning / DQN)：** 各行動がどれだけ「良い」か（Q値）を学び、その中から最高のものを選びます。
> 「左に押すスコアは7、右に押すスコアは5だ → 左に押そう！」

**新しい方法 (方策勾配法 / Policy Gradient)：** どの行動を「選ぶべきか」を直接学びます。
> 「ポールが右に傾いたら、80%の確率で右に押し、20%の確率で左に押そう。」
*（**勾配/Gradient** という言葉は、これらの確率を正しい方向に少しずつ調整するために踏み出す数学的な「ステップ」を指します。）*

**実生活の例：** 自転車の乗り方を学ぶ。
- 従来の方法：「左に5度傾く」vs「左に7度傾く」の*正確なスコア*を計算する。
- 新しい方法：体が覚えるまで練習する。しっくりくる方の足を踏み込む！

---

## REINFORCEはどのように機能するのか？

REINFORCEは、ロボットがゲームを最初から最後まで（1つの**エピソード**）プレイするのを観察し、その後こう問いかけます：「どの行動が良いスコアにつながったか？ それをもっとやろう！」

### ステップ・バイ・ステップ

**1. エピソードをプレイする**

ロボットは選択を行い、経験を収集します：
```
ステップ 1: 状態 = [ポールが右に傾いている] → 行動 = 右に押す → 報酬 = +1
ステップ 2: 状態 = [ポールがほぼ安定している] → 行動 = 右に押す → 報酬 = +1
ステップ 3: 状態 = [ポールが左に傾いている] → 行動 = 左に押す → 報酬 = +1
...
ステップ 47: 状態 = [ポールが倒れた！] → エピソード終了
```

**2. リターンを計算する**

各ステップについて、G_t、つまり「その時点以降の**合計報酬**」を計算します：
```
ステップ 47 における G = 1
ステップ 46 における G = 1 + 0.99 × 1 = 1.99
ステップ 45 における G = 1 + 0.99 × 1.99 = 2.97
...
ステップ 1 における G  = 47 (だいたい。最初から数えるのでリターンは高くなる)
```

γ = 0.99 の**割引率 (discount factor)** は、遠い将来の報酬の価値を少し低く見積もることを意味します。

**実生活の例：** 学校の初日に金色のシールをもらうのは、100日後にもらえるかもしれないと知っているよりもワクワクします。将来の報酬はわずかに「割り引かれ」ます。

**3. 方策を更新する**

取られた各行動について：
> もし G_t が高かった（その行動が良い結果につながった）：**それをもっとやろう！**
> もし G_t が低かった（その行動が悪い結果につながった）：**それは控えよう！**

数式： `loss = -log_prob(action) × G_t`

勾配をとって方策を更新することは、ロボットにこう伝えるようなものです：
*「ステップ20で取ったあの行動？ 次回はあと5%くらい頻繁にやってみよう！」*

## 方策ネットワークとは何か？

テーブルの代わりに、**ニューラルネットワーク**を使って方策を表現します。

```
観測状態          方策ネットワーク       行動確率
[カート位置]  →   [128 ユニット]  →  →  [左に押す: 30%]
[カート速度]  →   [128 ユニット]        [右に押す: 70%]
[ポール角度]  →
[ポール速度]  →
```

ネットワークは各行動の**確率**を出力します。そこからサンプリング（抽出）します：
> サイコロを振る → 1-30：左に押す、31-100：右に押す

**実生活の例：** 天気予報アプリが「降水確率70%」と言っています。雨が降ると決まったわけではありませんが、あなたはその確率に基づいて判断します。ロボットも同じことをします！

## 正規化：なぜ平均を引くのか

G_t を使って更新する前に、正規化（Normalization）を行います：
```
G_normalized = (G - mean(G)) / std(G)
```

**なぜか？** すべての報酬が正である場合を想像してください（CartPoleは常にステップごとに+1なので、まさにそうです）。正規化しないと、すべての行動が「良い」ように見えてしまい、更新の信号が混乱してしまいます。

正規化した後は、いくつかのリターンは正（平均以上 → もっとやる）になり、いくつかは負（平均以下 → 控える）になります。信号がはるかに明確になります！

**実生活の例：** 先生がテストを相対評価で採点します。平均点が70点であなたが85点なら、素晴らしいです！しかし、平均が90点であなたが85点なら、それは平均以下です。生の点数だけでは全体像は分かりません。

---

## 問題点：高い分散 (High Variance)

REINFORCEには大きな弱点があります。それは**分散 (variance)** です。リターン G_t は非常にノイズが多いのです。

**実生活の例：** 各レストランで一度食事をしただけで、そのシェフの腕前を判断することを想像してください。ある日はシェフの体調が悪かったかもしれませんし、ある日は食材が良くなかったかもしれません。一回の食事だけでは、そのレストランが良いかどうかを確実に知るには不十分です！

REINFORCEは更新の前にエピソードが完結するのを待ちます。あるエピソードはとても運が良く、別のエピソードはとても運が悪いかもしれません。勾配はあちこちに飛び跳ねます。

グラフの学習曲線がギザギザしているのはそのためです：
- ある実行では 500（最高！）
- ある実行では 50 に落ちる（最悪！）

ノイズが多いにもかかわらず、REINFORCEは最終的に学習します。しかし、忍耐が必要です。

## 結果

```
エピソード  100 | 平均報酬 (直近100回):  43.1
エピソード  200 | 平均報酬 (直近100回): 193.9
エピソード  500 | 平均報酬 (直近100回): 408.4
エピソード 1000 | 平均報酬 (直近100回): 500.0  ← クリア！
```

ロボットはポールを最大500ステップ保つことを学習しました。クリアです！

分散の問題があるにもかかわらず、CartPoleにおいてREINFORCEが効果的なのは以下の理由によります：
1. エピソードが短い（そのため、一回のトレーニングで多くのエピソードを得られる）
2. 最適な方策が単純である（主にポールが傾いた方向に押すだけ）

## 重要なポイント

| コンセプト | 平易な説明 |
|---------|---------------|
| **方策 (Policy)** | 行動を選択するためのロボットのレシピ |
| **エピソード** | ゲーム開始から終了までの一連の流れ |
| **リターン G_t** | 現時点からの将来の合計報酬 |
| **割引率 γ** | 将来の報酬を即時の報酬より少し低く評価すること |
| **正規化** | 信号を明確にするために平均を引くこと |
| **分散** | 勾配の推定値がどれだけあちこちに飛び跳ねるか |

---

## 次は？

REINFORCEの大きな弱点は**分散**です。次のスクリプト (`reinforce_baseline.py`) では、このノイズを劇的に減らす**ベースライン (baseline)** を追加します。これは、アルゴリズムが平均的に何を学ぶかを変えることなく、安定性を高めます。

コアとなるアイデア：「この行動は良かったか？」と聞く代わりに、「この行動は**期待以上に良かったか**？」と問うことです。この小さな変更が、学習をはるかに安定させます。
