# PPO для непрерывного управления: заставляем BipedalWalker ходить

## Дискретные и непрерывные действия

До сих пор в каждой среде, которую мы решали, были **дискретные** действия:
- CartPole: толкать ВЛЕВО или ВПРАВО (2 варианта)
- LunarLander: ничего не делать / лево / основной / право (4 варианта)

Но роботам в реальном мире нужны **непрерывные** действия:
- Человекоподобный робот: «с какой силой нажимать на каждый сустав» (любое значение от -1 до +1)
- Автомобиль: «на какой именно угол повернуть руль» (любой угол от -30° до +30°)
- Манипулятор: «приложить ровно 2.3 Ньютона в этом направлении»

**Пример из жизни:** Печать на клавиатуре = дискретные действия (нажать A, B, C...). Письмо карандашом = непрерывные действия (сместить руку на 2.3 см вправо, нажать с силой 40 г...).

## Гауссова политика для непрерывных действий

Для непрерывных действий вместо категориального распределения (выбор из N категорий) мы используем **нормальное (Гауссово) распределение**:

```
Действие ~ Normal(μ, σ)
```

Где:
- **μ (мю, среднее)**: Центр распределения — значение действия, к которому сеть «стремится»
- **σ (сигма, стандартное отклонение)**: Разброс — сколько случайности / исследования добавить

```
        Вероятность
             │
        0.4 ─┤      ██████
             │    ████████████
        0.2 ─┤  ██████████████████
             │████████████████████████
             └──────────────────────── Значение действия
           -1  -0.5   0   0.5   1
                      ↑
                   среднее μ
```

**Пример из жизни:** Опытный лучник целится в центр мишени (μ). Его стрелы не попадают в одну и ту же точку — есть некоторый разброс (σ). По мере практики он становится точнее (σ уменьшается), оставаясь нацеленным на «яблочко».

## Наша сеть Актер-Критик с Гауссовым распределением

```
Состояние (24 числа) → [256 нейронов] → [256 нейронов] →
    ├── Актер: 4 средних значения (μ₁, μ₂, μ₃, μ₄)
    │          + 4 параметра log_std (общие для всех состояний!)
    └── Критик: 1 значение (V(s))
```

Параметр `log_std` (логарифм **стандартного отклонения** — мера разброса или неопределенности) является **обучаемым параметром**, не зависящим от состояния. Это упрощает модель, позволяя уровню исследования меняться в процессе обучения.

**Почему log_std вместо std?** Стандартное отклонение должно быть положительным. Использование `log_std` позволяет сети выдавать любое вещественное число (положительное или отрицательное), а затем мы применяем `exp(log_std)` — экспоненциальную функцию, обратную логарифму, — чтобы получить гарантированно положительное std. Это предотвращает ситуацию, когда std становится отрицательным или нулевым.

## Вычисление логарифма вероятности для непрерывных действий

Для дискретных действий: `log_prob = log(P(action=LEFT))`

Для непрерывных действий нормальное распределение описывает плавную колоколообразную кривую вокруг среднего значения. Вероятность одного конкретного значения в непрерывной математике равна нулю, поэтому мы используем высоту кривой в этой точке, называемую **pdf** (функция плотности вероятности):
```
log_prob = Σᵢ log[Normal(μᵢ, σᵢ).pdf(aᵢ)]
```

`log` означает натуральный логарифм. Он превращает крошечные значения плотности в стабильные числа, которые нейронным сетям легче оптимизировать. Мы суммируем значения по всем измерениям действия (4 для BipedalWalker), потому что полное действие — это один вектор из 4 чисел.

**Пример из жизни:** Какова вероятность того, что завтра будет ровно 5.732...°C? Для непрерывной погоды вы посмотрите на кривую нормального распределения и увидите, насколько она высока в этой конкретной точке. Более вероятные температуры (близкие к среднему) имеют более высокую вероятность.

## BipedalWalker: Испытание ходьбой

BipedalWalker-v3 — это 2D-робот, который должен научиться ходить, не падая:

```
          O (голова)
         /│\
        / │ \
       /  │  \
      L   │   R   ← две ноги, каждая с коленным суставом
     / \  │  / \
    ●   ● │ ●   ●  ← 4 мотора (тазобедренный/коленный для каждой ноги)
```

**Пространство состояний (24 числа):**
- Корпус: угол, угловая скорость, горизонтальная скорость, вертикальная скорость (4 числа)
- Суставы: 4 мотора (2 тазобедренных, 2 коленных), каждый предоставляет угол и скорость, плюс 2 датчика контакта с землей (10 чисел)
- 10 датчиков LIDAR (показания расстояния до земли впереди) (10 чисел)

**Пространство действий (4 непрерывных значения, каждое в [-1, 1]):**
Значения действий управляют **крутящим моментом** (вращательной силой моторов) для ровно 4 суставов:
- Тазобедренный 1, Коленный 1, Тазобедренный 2, Коленный 2

**Награды:**
- +300 за достижение цели (правая сторона)
- -100 за падение (касание земли корпусом)
- Небольшая награда за каждый шаг продвижения вперед
- Небольшой штраф за использование двигателей (поощрение эффективности)

**Задача решена, когда:** Средняя награда > 300 за 100 эпизодов

## Ключевые отличия от дискретной PPO

Все то же самое, КРОМЕ:

| | Дискретная PPO | Непрерывная PPO |
|---|---|---|
| **Политика** | Categorical(logits) | Normal(μ, σ) |
| **Выборка** | действие = выбор из {0,1,...,N} | действие = μ + σ × шум |
| **log_prob** | log P(action=k) | Σ log Normal(μᵢ, σᵢ).pdf(aᵢ) |
| **Ограничение** | Не требуется | Ограничение (clamp) действий до [-1, 1] |

**Logits** — это необработанные, ненормированные оценки для дискретных действий. Категориальная политика преобразует их в вероятности с помощью **softmax** — функции, которая берет любой набор чисел и сжимает их в допустимое распределение вероятностей. Непрерывная PPO **не** использует softmax для самого действия, так как действие не выбирается из фиксированного списка. Вместо этого политика выдает среднее значение и стандартное отклонение нормального распределения.

**Clamp** означает принудительное удержание значения в допустимом диапазоне. Код использует `action.clamp(-1, 1)`, чтобы среда никогда не получала команды мотору за пределами допустимых границ.

**Clip** в PPO означает другое: PPO обрезает отношение вероятностей внутри функции потерь, как объяснено в [разделе об ограничении PPO](./ppo_scratch_explained.md#прием-с-ограничением-clipping). Ограничение действий (clamp) защищает интерфейс среды; ограничение PPO (clip) защищает обновление политики.

## Ходьба с нуля: Чему учится агент

**Начало обучения (отрицательные награды):** Робот беспорядочно дергается и тут же падает. Каждый эпизод заканчивается крушением через секунды.

**Середина обучения:** Робот обнаруживает, что поочередное движение ног создает продвижение вперед. Он начинает делать маленькие неуклюжие шаги — награда становится менее отрицательной.

**Конец обучения:** Формируется плавная и эффективная **походка**. Походка — это повторяющийся паттерн движений. Робот динамически подстраивается под неровности рельефа, используя свои датчики LIDAR для адаптации шагов в реальном времени.

**Пример из жизни:** Ребенок учится ходить:
1. Сразу падает (отрицательная награда)
2. Делает один шаг и падает (чуть менее отрицательная)
3. Делает несколько шагов (небольшая положительная награда)
4. Идет через всю комнату (большая положительная награда!)

## Почему BipedalWalker нужен PPO (а не REINFORCE)

- **Эпизоды BipedalWalker** могут длиться до 1600 шагов (намного дольше, чем CartPole!)
- **Награды разреженные** — вознаграждение за продвижение вперед крошечное на каждом шаге
- **REINFORCE потребовались бы** тысячи полных эпизодов, чтобы получить полезный сигнал

Обновления n-шагов PPO с **GAE (Generalized Advantage Estimation)** позволяют роботу учиться на неполных эпизодах:
> «Даже если я упал через 50 шагов, эти шаги показали НЕКОТОРОЕ продвижение вперед.
> Позвольте мне использовать оценку доходности за 50 шагов, а не ждать завершения эпизода».

## Результаты

После 500 обновлений (≈ 1 миллион шагов в среде):
- Робот демонстрирует видимый прогресс от случайных подергиваний к осмысленному движению вперед
- Стабильное улучшение кривой обучения
- Полная сходимость к награде > 300 требует большего обучения (5-10 млн шагов)

Кривая обучения показывает характерную «S-образную кривую» непрерывного управления:
1. Медленный начальный прогресс (стабилизация обучения)
2. Быстрое улучшение (открытие походки)
3. Постепенная доработка (оптимизация походки)

## Основные выводы

| Концепция | Простыми словами |
|-----------|------------------|
| **Гауссова политика** | Вместо выбора из меню — бросок дротика в диапазон значений |
| **μ (среднее)** | Куда «целится» политика |
| **σ (std)** | Сколько случайности / исследования использует политика |
| **log_std как обучаемый параметр** | Глобальный уровень исследования, обновляемый оптимизатором — как и любой другой вес сети |
| **Непрерывное управление** | Управление вещественными значениями на выходе (моменты силы, углы) |

## Что дальше?

У PPO много **гиперпараметров** — настроек, которые вы выбираете перед началом обучения (в отличие от параметров, таких как веса сети, которые изучаются автоматически). Примеры: `clip_eps`, скорость обучения, количество эпох и размер пакета.

Насколько PPO чувствителен к этому выбору? `ppo_hyperparams.py` запускает эксперименты, систематически меняя каждый гиперпараметр, и показывает эффект на скорость и стабильность обучения.
