# Using a Learned Model for Planning (MPC) 🔮

## The Big Idea {#core-idea}

You have a **world model** (a neural network that predicts the future).
Now what?

The most direct use is **planning**: at every moment, ask the model "what
would happen if I tried *this* plan? *that* plan? *that other* plan?"
Then do whichever plan looks best — but **only the very first step of it**.
Because the model isn't perfect, we execute just one action, observe the actual new state from the real environment, and then re-plan from scratch.

This trick has a name: **Model Predictive Control** (MPC).

---

## A Real-Life Analogy {#real-life-analogy}

You are at a restaurant looking at the menu. You don't commit to a five-course
order on the spot — you order the first course, see how full you feel, then
re-decide on dessert.

Or: you are driving on a winding road. You don't lock in steering inputs for
the next 30 seconds — you constantly look ahead, plan a few seconds out, take
the next steering action, and re-plan.

That **plan-far / act-near / re-plan** loop is MPC.

---

## How "Random Shooting" Works {#how-random-shooting-works}

There are more sophisticated planners — for example:
- **CEM** (Cross-Entropy Method): iteratively refine a distribution over plans by keeping only the top-k each round.
- **MCTS** (Monte Carlo Tree Search): build a search tree guided by simulation statistics, used by AlphaGo and MuZero.
- **Gradient-based planners**: differentiate the model's predictions with respect to the actions and follow the gradient directly.

We use the simplest that works: **random shooting**.

```
Given the current state s:
    1. Sample N=200 random action sequences of length H=15.
    2. For each sequence, simulate it through the world model from s, summing
       a shaped reward at each step.   (200 dreams in parallel — fast!)
    3. Find the sequence with the highest predicted total reward.
    4. Execute that sequence's FIRST action in the real environment.
    5. Observe the real next state.  Discard the rest of the plan.
    6. Go to step 1 — re-plan from scratch.
```

200 plans × 15 steps = 3,000 imagined transitions per real step. The world
model runs them all in a single batched neural-network forward pass — typically
a few milliseconds.

---

## Why Re-plan Every Step? {#why-replan-every-step}

Because the model is imperfect. Errors compound over a rollout (as seen in the chart generated by `world_model.py`, saved in `outputs/world_model.png`). The plan at step 0 is reliable only for the first few moves; by step 15 the model is hallucinating. So we trust only the
**first move**, then refresh the plan with the latest real state.

This is the same reason humans don't write a 100-move chess plan and stick to
it — circumstances change, and the further out you guess, the less it matches
reality.

---

## A Wrinkle: The Reward Has to Tell the Planner Something {#smooth-rewards}

In CartPole the real reward is `+1` every step until the pole falls. The model
will faithfully predict `+1, +1, +1, ...` for almost every plan, because
random plans rarely terminate quickly inside the model — and so every plan
scores the same. The planner has nothing to choose between.

The fix: replace the real reward with a **smooth proxy** during planning:

```python
reward_proxy(state) = 1
                    - |pole_angle| / 0.21          # pole upright?  (1=yes)
                    - 0.1 * |cart_position| / 2.4  # cart centred?  (1=yes)
```

Now plans that *would* end with a fallen pole get visibly worse scores than
plans that stay upright. The planner can rank them.

> **Real-life lesson.** A flat reward signal — "you survived another second" —
> is useless for short-term planning. Dense, shaped signals help.

## What Our Code Does {#what-our-code-does}

`model_based_planning.py`:

1. **Loads** the world model weights saved by `world_model.py`. (If they're
   missing, it retrains one on the fly.)
2. **Runs 20 episodes** of MPC on the real CartPole-v1.
3. **Also runs 20 episodes** with a uniformly-random policy, as a baseline.
4. **Plots** both side-by-side and prints averages.

## Strengths of Model-Based Planning {#advantages}

- **Sample efficient.** All the learning was done from one batch of random
  transitions. No further env interaction was needed to derive a useful policy.
- **Easy to retarget.** Want to control the agent differently? Change the
  reward proxy — no retraining. (Try maximising cart velocity for fun.)
- **Interpretable.** You can inspect the plans the agent considered, the
  predicted trajectories, and the scores.

## Weaknesses (and What People Do About Them) {#limitations}

- **Random shooting is dumb.** It samples plans blindly. For higher
  dimensions, switch to **CEM** (Cross-Entropy Method — see above) or
  **iLQR** (Iterative Linear-Quadratic Regulator, a classic optimal-control
  method that approximates the model as locally linear and solves it
  analytically) or full **gradient-based** planner improves actions by following gradients through a
  differentiable model.
- **Compounding model error.** Long horizons drift. People use **probabilistic
  ensembles** (several models trained on the same data, as in PETS, Chua et
  al. 2018) so the planner can notice disagreement and penalise plans the
  model is uncertain about.
- **Real reward is what we want, ultimately.** Reward shaping helps, but for
  more complex tasks people learn a **value function** trained *inside* the
  world model — a learned critic that estimates the long-run return from any
  state without requiring a hand-crafted proxy. Both **Dreamer** (which trains
  an actor-critic entirely in latent imagination) and **MuZero** (which pairs
  MCTS with a learned value network) use this idea.

## How This Connects to Modern Systems {#modern-systems}

The exact recipe you just ran — **learned dynamics + planning** — is the
backbone of some of the strongest RL systems in modern AI research:

- **MuZero** (DeepMind): combines a learned world model with Monte Carlo Tree Search. It mastered Go, chess, shogi, and Atari without needing the rules beforehand.
- **Dreamer / DreamerV3** (Hafner et al.): trains a policy *inside* a learned
  **latent-space** world model (meaning the model compresses raw images or states into a compact, abstract representation before predicting the future). It achieves state-of-the-art (top-tier) performance across 100+ benchmarks.
- **PETS / PlaNet / TD-MPC**: these are families of algorithms (research lines) that scale exactly this idea to complex continuous control tasks like robotics.

You have built — in a few hundred lines — the smallest member of that family.

## Key Words {#key-terms}

| Term | Plain English |
|------|---------------|
| **MPC**           | Model Predictive Control — plan ahead, act once, re-plan |
| **Random shooting** | Score lots of random plans, pick the best |
| **Horizon (H)**   | How many steps the plan looks ahead |
| **N samples**     | How many candidate plans we consider per step |
| **Receding horizon** | Re-planning at every step instead of committing to one plan |
| **Reward proxy / shaping** | A smooth surrogate reward that gives the planner a useful signal to optimize |

## One-Sentence Summary {#summary}

> **Once you have a world model, planning is just "dream a hundred futures,
> pick the best first step, repeat."**

That's the whole secret of model-based RL.
